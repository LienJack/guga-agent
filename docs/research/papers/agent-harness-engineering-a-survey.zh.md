## 1 简介

### 1.1 绑定约束：模型上的Harness

基于LLM的智能体的学术研究总的来说是对模型的研究。研究议程集中在模型可以做什么：是否可以跨多个步骤进行规划、可靠地调用工具、检索和压缩相关记忆，或者与其他智能体进行协调。隐含的假设是 智能体能力主要是模型能力的函数，具有足够好的提示的足够能力的模型将产生足够可靠的行为。

最近的经验证据挑战了这样的假设：更好的模型本身就能产生更可靠的智能体。最近的三个结果确立了这一模式。 Bölük (2026a) 仅修改了编辑工具格式和周边工具Harness，没有模型修改，并报告在 15 个模型的编码基准上获得了高达 10 倍的增益。 Trivedy (2026) 仅通过系统提示重组、中间件上下文注入和自我验证挂钩，就将 Terminal-Bench 2.0 上的固定 GPT-5.2-Codex 智能体从 52.8% 提高到 66.5%，13.7
百分点的增长完全是通过基础设施变革实现的。  元-Harness (Lee et al., 2026)
通过自动 Harness 优化，在 Terminal-Bench-2 上实现了 76.4%，超越了所有手工设计的
方法无需修改模型权重。  在每种情况下，变量都是执行Harness（
管理上下文构建、工具交互、编排、反馈和执行约束的基础设施层）；模型保持固定。这些仅 Harness 的收益均超过了典型的 2 到 4 个百分点的改进，这些改进被报告为在相同基准上有意义的模型进步。该模式并非偶然：推动结果的是Harness，而不是模型。

我们将此模式称为绑定约束论文 (Bölük, 2026b)：对于跨可比前沿模型评估的长期任务，基准方差可能由执行 Harness 驱动，也可能由模型本身驱动。我们使用这篇论文作为调查其余部分的框架。

§2 中的图 3 说明了历史上的相同转变：早期系统将功能集中在单个模型循环中，而后来的系统越来越多地将可靠性暴露为跨层基础设施问题。

### 1.2 实践者与研究人员之间的差距

从业者的紧迫性和研究词汇之间存在着紧张关系。 OpenAI 明确将“Harness 工程”定义为围绕 Codex 智能体设计环境、约束、文档和反馈循环的学科，并于 2026 年 2 月报告称，一个小团队在五个月内生产了大约 100 万行的内部产品，而无需手动编写生产代码(OpenAI, 2026a)。 Anthropic 的 智能体工程帖子从相邻的方向得出了相同的原则：有效的 智能体应该使用简单且可检查的架构，工具接口应该设计用于 智能体使用而不是从面向人类的 API 复制，上下文应该逐步公开而不是急切地加载，长期运行的工作需要持久的切换工件和可恢复的执行基础设施 (Anthropic, 2024a; Aizawa, 2025; Anthropic Applied AI Team, 2025; Anthropic, 2025d; 2026b)。 Martin Fowler 网站上的一篇文章将 Harness 工程描述为“AI 智能体的控制论调节器”，由前馈指南和反馈传感器组成，围绕 LLM (Böckeler, 2026) 形成控制回路。

与此同时，研究界一直在越来越精确地研究 智能体系统的组成部分：内存、工具使用、规划和安全性。尚未系统研究的是如何将这些组件集成到可靠运行的系统。结果是从业者与研究人员之间存在差距：从业者知道Harness基础设施很重要，但缺乏正式的词汇来描述其原因，以及能够实现系统改进的术语。这项调查试图弥合这一差距。

3. 主张 3（实证）：将 148 多个开源项目映射到 ETCLOVG 上，显示生态系统的哪些区域密集、哪些区域薄弱，以及早期语料库遗漏了哪些类别。该映射是迄今为止最大的开源 智能体-Harness 语料库。执行、工具、生命周期和验证都被密集覆盖；可观察性和治理更薄，更多地存在于商业平台中；早期语料库中没有的三个类别，包括任务运行器、多智能体协调器和规范驱动的开发工具，现在都是一流的。方法小节（附录 A §2.4-§2.9）使编码可重复，并且映射支持§3-§9 中的逐层观察。

### 2.2 三个工程阶段

2022-2026 年期间揭示了该领域选择设计的连贯的三相演变。

提示工程(2022-2024)。主要杠杆是输入提示文本。从业者通过制定更好的说明、少量示例和推理模板来进行优化。工程范围很窄：将单个文本输入优化为单个模型调用。

上下文工程(2025)。随着 智能体运行时间变得更长，绑定约束从“输入是什么？”转变为“输入是什么？”到“模型在每一步应该看到什么信息？”此阶段的重点是上下文管理：每轮注入什么，如何检索和压缩内存，如何按相关性对工具结果进行排名，以及如何处理上下文窗口饱和。范围从单个输入扩展到管理流入上下文窗口(Anthropic Applied AI Team, 2025)的多个信息流。

Harness工程(2026)。随着模型变得足以处理长时间运行的任务，可靠性越来越依赖于维护状态、协调工具、注入反馈、强制约束和验证进度的基础设施包装器。这一观察结果与绑定约束观点一致，即长期智能体性能是由耦合模型-Harness系统产生的，而不是由模型单独(Bölük, 2026b)产生的。因此，Harness 工程提出必须围绕模型设计哪些治理、约束、反馈循环和执行控制，以使智能体 系统可靠。在我们的分类中，此阶段将所有七个 ETCLOVG 层视为一个集成的整体(OpenAI, 2026a; Böckeler, 2026; LangChain, 2026b)。

每个阶段都包含前一个阶段： Harness 工程包括上下文工程，其中包括提示工程。这三个阶段在时间和概念上也有重叠，而不是通过清晰的界限相继出现。即时工程仍然是当今 Harness 实践和上下文工程的活跃部分

### 2.3 ETCLOVG 七层分类法

我们为智能体 Harness 工程提出了七层分类法。我们用缩写 ET-CLOVG 来称呼它，它代表执行、工具、上下文、生命周期、可观察性、验证、治理。图4给出了紧凑的可视化图；本小节确定了整个调查中使用的解释。

前四层描述了Harness的结构核心。执行 (E) 确定 智能体代码运行的位置以及绑定它的沙箱约束；工具（T）指定如何描述、发现和调用外部能力；上下文 (C) 控制模型在短期、会话级别和持久范围内可以看到的内容；生命周期 (L) 组织读取和写入该状态的控制流，从单智能体循环到多智能体和问题到拉请求工作流程。其余三层描述了该核心周围的控制平面。可观察性 (O) 捕获痕迹、成本、故障和可靠性信号；验证（V）将任务和痕迹转化为评估、失败归因和回归反馈；治理 (G) 通过许可、身份、策略、强化、审计和人工监督机制来约束行为。第 3-9 节按顺序开发这七个层，而第 10-12 节综合了跨层权衡和不属于任何单层的开放问题。

这种分类法有两种设计选择。  首先，我们将可观察性（O）提升为独立的
层而不是将其视为生命周期挂钩的副作用。在生产系统中，可观测性拥有专用的工具生态系统（Langfuse、Arize Phoenix、OpenLLMetry）和独特的工程实践（OpenTelemetry 仪器、成本归因、异常检测），保证独立处理。其次，我们引入治理（G）作为一流的层，它捕获了全方位的安全性和安全性。

### 2.4 范围

我们使用 智能体 Harness 的含义比“围绕 LLM 的任何软件”更狭义：Harness 是工程包装器，它通过执行基础、工具接口、上下文控制、编排、可观察性、评估反馈和治理约束 (OpenAI, 2026a; Anthropic, 2025d; LangChain, 2026b) 将模型调用转变为有界的、有状态的、工具介导的任务执行。因此，分析单元是使长期运行的 智能体行为可控、可检查和可恢复的基础设施，而不仅仅是基础模型或提示。我们根据功能而不是产品类别来划分边界：当 智能体框架公开可重用机制（例如状态编排、工具路由、运行时策略挂钩或跟踪捕获）时，它就在范围内；瘦模型 API 包装器、提示库、静态数据集、通用容器运行时、矢量数据库、APM 仪表板或内容过滤器超出范围，除非明确适应 智能体执行、状态、评估或工具使用治理。

### 2.6 纳入和排除标准

当一个项目满足三个条件时就被纳入：它已公开记录、已实施或
指定了具体的Harness级机制，并且现有证据足以分配至少
一层 ETCLOVG 层。  这包括具有可重用编排或工具路由逻辑的智能体框架，
实例化可执行智能体环境的基准、为智能体执行打包的沙箱，以及在智能体状态、跟踪、操作或策略上运行的内存、可观察性、评估或治理系统。我们排除了简单的聊天机器人演示、提示包、瘦模型客户端包装器、没有智能体运行时的静态数据集或排行榜、不面向智能体的通用基础设施组件，以及无法从公共文档中检查其技术行为的产品页面。边界案例是通过机制而不是标签来解决的：名为“智能体”的存储库不足以包含在内，而当评估或沙箱项目提供可重复使用的Harness机械时，则包含在内。

### 2.7 编码协议

每个保留的项目都使用公共工件本身作为证据，针对七个 ETCLOVG 层进行编码：自述文件、文档页面、论文、示例、发行说明，以及需要时的存储库结构。编码是多标签的，因为许多系统跨层；主要层标记了工件最核心的机制，而仅当文档公开独立功能而不是偶然依赖项时才分配次要层。当前快照使用带有作者审核的单主编码器协议，而不是正式的多编码器协议研究，因此我们不报告 Cohen 的 kappa 或可比较的注释者间统计数据。在对全套代码进行编码后，使用保守规则重新审视不明确的案例：如果公共证据没有明确显示智能体面向机制，则保留层分配。

## 3 执行环境和沙箱（E）

我们通过执行环境和沙箱 (E) 层展开逐层处理，这是 ETCLOVG 七大支柱中的第一个（§1 中的权利要求 2）。我们讨论的系统来自支持权利要求 3 的 170 多个项目语料库。

### 3.1 范围和概念

### 3.1.1 定义

智能体 的执行环境是指物理执行智能体 操作的基础设施层。在LLM 智能体的上下文中，执行环境和沙箱是紧密耦合的概念。因此，生产智能体系统几乎总是在沙盒环境中执行操作。

### 3.1.2 为什么沙盒是 智能体时代的核心

智能体时代的沙箱不仅仅是从传统多租户代码执行继承的安全措施。它同时服务于三个不同的目的，这三个目的的结合将沙箱从操作细节提升为 智能体 Harness 设计中的首要关注点。

第一个目的是安全。 智能体沙箱面临着传统多租户代码执行之外的挑战。 LLM 生成的代码既不可审计，也不可大规模预测，因此排除了静态审查作为主要防御措施的可能性。 智能体在多个步骤中自主执行，因此在操作执行时无法进行人工干预。即时注入攻击可以将原本良性的 智能体重新用作沙箱定向攻击的向量，模糊可信用户意图和恶意输入之间的界限。最近关于沙箱逃逸的实证研究表明，这些担忧并不是假设的，我们将定量证据推迟到第 3.3 节。

第二个目的是可重复性。长期 智能体任务和测量它们的 评测 Harness (§8) 需要能够将执行状态重置为已知基线。 Docker 容器或 microVM 可以根据需要销毁和重建，而开发人员的工作站则不能，而这一特性正是基于沙箱的评估标准（例如 SWE-bench Jimenez 等）的原因。 (2024) 和 OSWorld Xie 等人。 (2024) 实用。在训练时，当单个任务可能在并行轨迹上重播数百次时，缺乏廉价的重置机制本身就是一个可扩展性瓶颈。

第三个目的是活跃性，这是智能体时代最具体的一个目的。如果没有沙箱，智能体 希望执行的每个潜在风险操作，例如文件写入、程序包安装或

<!-- page 12 -->

出站网络呼叫必须通过向人员发出明确的权限提示来进行控制。从规模上看，这会产生两种失败模式：用户出于沮丧而放弃智能体，或者他们条件反射地批准一切并违背提示的安全原理。沙箱通过定义一个有界区域来打破这种僵局，智能体 被授权在该区域内自由行动，将权限从操作问题转移到会话配置。 Anthropic 报告称，在 Claude Code 中引入沙箱可以减少 84% 的权限提示，同时保持安全 Anthropic (2025b)。在这个框架下，沙箱同时是一个笼子和一个许可证，而许可证方面首先是实现自主长期执行的原因。

在这三个目的中，安全性与传统沙箱共享，可重复性在 智能体设置中得到放大，而活性本质上是新的。它们的结合证明了将 智能体沙箱视为一个独特的研究对象，而不是容器技术的下游应用。

### 3.2 智能体沙箱的类别

智能体 沙箱基础设施在 2024 年到 2026 年期间已经多样化，从一小组通用运行时到几个不同的产品类别，每个类别都针对不同的 智能体任务类型进行了优化。我们沿着工作负载和用例的轴将景观分为七个类别，我们发现这对于 Harness 设计师在系统中进行选择最有帮助。正交轴隔离技术，包括容器、用户空间内核（例如 gVisor Google (2018)）、microVM（例如 Firecracker Agache 等）。 (2020) 和 Kata 容器 Kata 容器项目 (2017)、WebAssembly 和操作系统级原语（例如 bubblewrap 和 Seatbelt）在每个小节中作为设计属性而不是顶级类别进行讨论，因为相同的隔离原语在不同的工作负载中重复使用。这七个类别是通用托管沙箱（§3.2.1）、计算机使用智能体基础设施（§3.2.2）、代码专用沙箱（§3.2.3）、框架集成运行时（§3.2.4）、浏览器评估环境（§3.2.5）、操作系统级权限沙箱（§3.2.6）和沙箱抽象层（§3.2.7）。以下小节介绍了每个类别。

### 3.3 威胁模型和沙箱逃逸

沙盒智能体 执行面临着传统的容器级威胁（容器逃逸、侧通道、资源耗尽）和放大经典问题的智能体 特定威胁类别。首先，提示注入攻击允许外部输入（例如检索的网页、工具响应或文件内容）劫持智能体行为并启动恶意沙箱操作。其次，目标不一致可能会导致智能体本身主动尝试沙箱逃逸作为工具性子目标。第三，当具有多个工具访问权限的智能体允许单个沙箱弱点跨集成功能级联时，就会发生组合放大。

关于 智能体设置中沙箱逃逸的经验证据有限，但令人担忧。   沙盒-

### 3.4 部署方式

智能体 沙箱基础设施在部署模式上已经多样化，超越了原始的自托管 Docker 模式。当前实践中三种模式并存。在自托管模式下，开发人员直接管理沙箱基础设施，这是OpenHands和SWE-agent的默认模式。在云 (SaaS) 模式中，沙盒作为服务提供商处理基础设施，例如 E2B、Modal 和 Daytona Cloud。在混合或自带云（BYOC）模式下，智能体逻辑和沙箱执行在环境中解耦；示例包括 OpenHands SDK 的本地和远程工作空间抽象 Wang 等人。 (2025c) 以及 E2B 和 Northflank 的 BYOC 产品。

这些模式的演变是由两种互补的力量驱动的。一方面，从业者报告沿着延迟、安全性和可扩展性三个轴制定部署选择 Anthropic (2025b;f)。自托管沙箱提供最低的延迟和最紧密的迭代循环，但承担全部运营负担；云沙箱扭转了这种权衡，以网络往返为代价提供弹性扩展和托管安全性；混合模式尝试将敏感数据保留在本地，同时委派执行能力。另一方面，数据驻留、合规性和可审核性等组织限制将部署推向混合架构，即使延迟和可扩展性有利于单一模式解决方案。

从观察的实践来看，自托管沙箱在交互式开发和单租户场景中占主导地位，而云沙箱在多租户和大规模部署中更常见。混合模式专门针对合规性或数据局部性要求与大量临时执行能力需求共存的场景而出现。在实际 智能体工作负载下，这些模式之间的系统实证比较仍然不存在，我们将其视为第 12.1 节中更广泛的运行时议程的一部分。

### 3.5 总结

执行环境是 智能体 Harness 的物理基础：它们提供安全边界、用于可重复评估和训练的重置机制，以及长期 智能体可以在没有指挥人员批准的情况下采取行动的有界区域。本节调查的七个类别表明，设计空间现在不再是由单一隔离原语塑造，而是由工作负载保真度、威胁模型、
和运营模式。  托管沙箱和计算机使用环境强调强隔离
和现实的执行力；代码专用和浏览器环境强调吞吐量和评估结构；框架集成的运行时捆绑功能以方便使用；操作系统级权限沙箱缩小本地权限；抽象层使基板可更换。

这种设计空间在能力、控制和成本之间造成了反复出现的紧张关系。高风险工作负载推动了微虚拟机和托管云的发展，交互式本地工作流程推动了轻量级权限边界的发展，大规模培训或评估则推动了快速可重置基底的发展。因此，关于运行时防御、经济性、可移植性、捆绑与组合设计以及跨层耦合等未解决的问题并不是孤立的执行层脚注；它们在§12.1 中作为调查范围内的开放问题重新出现。

### 4.1 协议和接口标准

MCP 已成为编码和企业智能体 最明显的工具集成基础，具有显式的主机-客户端-服务器架构和基于 JSON-RPC 的工具、资源和提示(Model Context Protocol, 2025b;c;a) 的类型化交换。 MCP 的实用价值不仅在于模式级互操作性，还在于生态系统的流动性：智能体 构建者可以重用扩展的服务器目录，而不是为每个部署实现自定义连接器。

A2A 的目标是不同但相邻的边界。它不是将工具暴露给一个智能体进程，而是标准化了不透明智能体式应用程序之间的通信，包括通过智能体卡进行发现、支持同步和流式交互以及长时间运行的任务协作(A2A Project, 2025)。最近的协议调查将 MCP 和 A2A 置于互补的角色：MCP 主要用于工具/上下文访问，A2A 用于 智能体间的委派和协作 (Ehtesham et al., 2025)。我们认为，一个更有用的组织原则是根据工具/接口标准跨越的集成边界而不是供应商沿袭或发布时间线对工具/接口标准进行分类。在这个镜头下，出现了四个边界：模型↔函数（结构化调用）、智能体↔外部能力（运行时到工具解耦）、智能体↔智能体（跨进程委托）和智能体↔Repo/环境（版本控制策略）。表 1 总结了这一观点，并明确了为什么几个广泛比较的标准（例如，MCP 与 A2A，或函数调用与 OpenAPI）在单个 Harness 中占据不重叠的角色。

### 4.2 工具描述、发现和选择

一旦协议定义了如何进行调用，下一个瓶颈就是在每个步骤中应该显示和选择哪些工具。越来越多的工作研究工具文档质量、检索和动态候选修剪。 EASYTOOL 分析了从大量库存中选择合适工具的挑战(Yuan et al., 2025)。 AnyTool 和 CRAFT 专注于通过自动构建或完善工具使用管道 (Du et al., 2024; Yuan et al., 2023) 来减少手动规范负担。 MetaTool 基准式评估表明，工具检索和调用质量在不同领域和查询形式之间存在很大差异(Huang et al., 2023)。 MCP-Zero、ToolRet 和 ToolRegistry 等最近的工作强调检索感知编排和注册表质量作为下游智能体 成功(Fei et al., 2025; Shi et al., 2025b; Ding, 2025) 的一阶决定因素。一个密切相关的方向将工具选择扩展到可重用技能，其中 智能体必须识别相关的程序模块，而不仅仅是紧凑的 API 模式。技能路由器（郑

<!-- page 18 -->

et al., 2026）和 SkillRet (Cho et al., 2026) 都大规模研究了这个技能选择问题，强调需要从大型且重叠的技能库中检索正确的技能。

在系统层面，这些结果强化了两个设计原则。首先，“更少但更好的工具”通常优于暴力工具暴露，因为它缩小了即时熵和规划器分支。其次，发现管道必须具有适应性：静态的全局工具列表无法扩展到快速发展的存储库或多租户企业部署。

### 4.3 工具增强训练和集成

第三个方向从运行时编排转向模型能力获取。 Toolformer 演示了自我监督增强功能，用于学习何时以及如何在生成 (Schick et al., 2023) 期间插入 API 调用。 Gorilla 和 ToolLLM/ToolBench 通过更大的工具语料库、指令调整管道以及以执行为中心的 API 使用监督(Patil et al., 2024b; Qin et al., 2024) 扩展了这条线。 ToolkenGPT 和 CREATOR 探索令牌级或控制器式集成，以提高调用格式化保真度和规划稳定性(Hao et al., 2023; Qian et al., 2023b)。

在生产Harness中，这些模型方面的进步通常与框架级运行时堆栈配对，例如LangChain、语义内核和smolagents，它们提供内存抽象、路由中间件和工具适配器(LangChain, 2026c; Microsoft, 2026; Hugging Face, 2026)。编码 智能体设置还公开了第二类更具语义的工具：静态分析器、类型检查器、求解器支持的验证器、证明助手以及补丁等效或故障定位检查器。 Ugare 和 Chandra (2026) 将这个空间构建为 智能体式 代码推理：智能体 探索存储库并推理代码行为，而不必执行代码。他们的半形式推理方法介于非结构化思想链和完全形式化验证之间，通过强制 智能体陈述前提、跟踪程序路径并得出明确的结论，从而改进补丁等效性验证、故障定位和代码问题
回答。  对于工具层来说，意味着自动推理工具应该返回证据
承载工件，例如痕迹、证明义务、反例或结构化证书，而不仅仅是黑匣子是/否答案。综合证据表明，工具能力是由预训练和微调信号、界面模式质量和运行时选择策略共同决定的；仅改进一个组件的收益有限。

### 4.4 可扩展性和会话管理

一个反复出现的运营挑战是长期的会话管理。有状态的工具会话提高了连续性，但增加了簿记复杂性，特别是当调用并行化或跨委派时
多个智能体。  故障模式包括过时的句柄、重试期间不一致的工具状态以及上下文
详细工具轨迹的窗口饱和度。  因此，有效的Harness设计需要明确的生命周期
对工具会话的控制、有界工具上下文注入以及可将错误归因于规划器逻辑或接口/协议故障的可观察性挂钩。

最后，生态系统级别的景观管理可以帮助从业者快速映射可用的协议和工具选项，但基于基准的比较对于决定在给定部署中应采用哪种协议和工具路由策略仍然至关重要。

### 5.1 为什么必须设计上下文

更大的上下文窗口并不能解决内存问题。  了解原因需要查看
架构。

二次注意力成本。 Transformer 的自注意力机制计算上下文 (Vaswani et al., 2017) 中每个标记之间的成对关系。对于 n 个标记，这会产生 n2 个成对权重；计算和内存规模与上下文长度呈二次方关系。 FlashAttention 和位置编码插值等工程技术减少了常数因子，但二次结构是架构性的。
将上下文加倍并不会使成本加倍，而是使其成本增加四倍。  这使得上下文窗口变得稀缺
资源。

U 形注意力曲线。仅靠架构成本并不能解释为什么模型在长上下文中失败，即使计算是可以承受的。刘等人。 (2024) 提供了关键的实证结果：在使用 20 个输入文档进行多文档问答时，当相关文档位于上下文中间时，与将其放置在开头或结尾相比，准确率下降了 30% 以上。这种 U 形性能曲线适用于模型、任务和上下文长度，包括专门在长上下文上训练的模型。实际意义是直接的：信息放置与信息存在同样重要。 智能体检索了正确的内容，但定位不佳，从检索中获得的收益很少。

上下文大规模腐烂。洪等人。 (2025) 在旨在将输入长度与任务难度隔离的受控条件下评估了 18 个前沿模型，包括 GPT-4.1、Claude Opus 4、Gemini 2.5 和 Qwen3。随着输入的增加，每个模型都会退化。退化是不均匀的并且是特定于任务的。语义模糊的查询（其中相关段落与问题在词汇上不匹配）显示出比精确匹配查询更严重的退化，指出了复合故障：模型无法本地化相关信息，

<!-- page 20 -->

一旦找到就无法对其进行推理。降解也会在窗户满之前就开始了。额定为 200K 代币的模型可能会在 50K 代币时表现出显着的性能损失。这一现象，令洪
等人。 (2025) 调用上下文腐烂，不是边缘情况。  这是任何 智能体的正常操作条件
通过多个步骤累积工具结果、中间推理和文件内容。

总而言之，这些结果表明上下文管理不能是事后的想法。关于包含内容、放置位置以及何时删除的每个决定都会对 智能体可靠性产生直接影响。

### 5.2 从即时工程到情境工程

从即时工程到情境工程的转变反映了(Karpathy, 2025)范围的转变。即时工程优化了单个模型调用的大部分静态文本输入。在视觉中，相同的框架扩展到连续的可学习标记：视觉提示调整将一小组可训练向量预先添加到冻结视觉变压器的补丁序列中，使其适应下游任务，同时更新不到 1% 的模型参数(Jia et al., 2022; Xiao et al., 2025)。文本和视觉变体都具有提示工程的核心特征：针对一种输入模式的固定、单次调用优化。上下文工程优化了多步骤任务中每个推理步骤中模型可用的完整信息状态。

Anthropic 的应用人工智能团队将其定义为“在 LLM 推理期间策划和维护最佳标记集的策略集，包括提示之外可能出现的所有其他信息”(Anthropic Applied AI Team, 2025)。接下来的指导原则是找到最小的高信号令牌集，以最大化每一步所需结果的概率。这个原则
驱动本节中的每项技术。  渐进式披露及时加载信息，而不是
预先。压缩会删除已达到其目的的令牌。内存检索仅提取与当前任务最相关的记录。

上下文包含什么内容。在部署的智能体中，推理时的上下文包括系统提示和行为指令、工具定义和模式、之前的转弯历史记录、当前轨迹的工具调用结果、检索到的文档或内存记录以及任何动态注入的工作状态。所有这些都在争夺相同的有限注意力预算。上下文工程意味着在每个步骤中对每个组件做出明确的选择，而不是让它们不受控制地累积。

这种做法的成熟在生态系统中是可见的。第一性原理手册 (Kimai, 2025) 和策划调查 (Meirtz, 2025) 现在将情境工程本身视为一门学科。三层内存架构，涵盖活动上下文窗口、会话级状态和跨会话存储，
成为占主导地位的组织框架。  它直接映射到经典的内存层次结构
操作系统，它提供了有用的直觉和词汇，用于将正确的技术与正确的时间范围相匹配。

### 5.3 短期：管理活动上下文窗口

短期上下文管理控制模型在单个推理步骤或会话中的短步骤序列中看到的内容。这些决策对模型行为具有最直接的影响，如果决策不当，成本也会最高。

系统提示校准。系统提示建立智能体的行为范围，并在每次呼叫上消耗固定的预算。人类应用人工智能团队(2025)将设计挑战描述为找到正确的高度：过于具体的提示会引入脆弱的、需要大量维护的逻辑；过于模糊的提示会使模型没有具体的指导。在实践中，有效的系统提示被组织成清晰划分的背景、说明、工具指南和输出格式部分，并使用 XML 标签或 Markdown 标头将它们分开。推荐的工作流程是从最佳可用模型的最小提示开始，凭经验识别故障模式，然后添加有针对性的指令来解决特定差距。先发制人地枚举每个边缘情况往往会使提示变得臃肿，而不会提高可靠性。

代币高效的工具设计。工具定义是主要的上下文使用者。每个架构、名称、描述和参数类型都会在每次调用时注入。在 智能体读取用户请求之前，大型工具集可能会消耗数万个令牌。生产部署中出现的原则是

### 5.5 长期：持久内存系统

长期内存系统提供索引的、可检索的存储，该存储在会话、任务和智能体实例中持续存在。中期技术依赖于摘要的前向注入，而长期系统支持任意调用：给定查询，系统检索最相关的存储记忆，无论它们何时创建。这一层是 智能体随着时间的推移积累经验的地方。

### 5.5.1 基础架构

受操作系统启发的分层内存。帕克等人。 (2023) 通过将 LLM 的上下文窗口视为 RAM，将外部存储视为磁盘，引入了关键抽象，并提供模型函数调用，使其显式地将信息分页进出。与虚拟内存的类比是精确的：正如操作系统通过透明分页为应用程序提供更大内存的幻觉一样，MemGPT 通过透明内存管理为 LLM 提供更长上下文的幻觉。 智能体可以在远远超出物理窗口限制的上下文上运行，按需从外部存储中检索相关历史记录。本文明确了 智能体内存和作为大多数后续长期内存系统基础的操作系统内存之间的联系。

观察、反思和检索。帕克等人。 (2023) 在社交模拟 智能体的背景下引入了内存流架构。内存流将 智能体的所有观察结果存储为带有时间戳和重要性分数的自然语言记录。在每个步骤中，检索模型都会通过组合三个信号来显示最相关的记忆：新近度、重要性（在写入时由 智能体评分）以及与当前查询的相关性。第二种机制是反思，定期将存储的观察结果综合成更高层次的见解。 智能体从最近的经验中询问自己最突出的问题是什么，检索相关记录，并生成存储回内存流的结论。消融实验表明，观察、反思和检索这三个组成部分中的每一个对行为质量都有独立的贡献；去除任何一个都会产生可测量的退化。观察反射-检索三元组仍然是智能体内存设计的标准模板。

动态遗忘和人格建模。  钟等人。 (2024) 在之上构建了 MemoryBank
添加了两个内容的检索架构。  第一个是动态遗忘机制，灵感来自

### 5.6 长期技术：在 100 多个回合中保持智能体 连贯

前面的每一节都讨论一个时间范围。长期任务，包括大型代码库迁移、多会话研究项目和扩展的自主工作流程，需要同时协调所有三个层，并实时决定在每个点应用哪种技术。本节介绍了这种规模出现的集成级技术。

上下文压缩。压缩总结了接近其极限的上下文窗口，并使用累积状态的压缩表示重新启动执行。 Anthropic Applied AI 团队 (2025) 详细描述了校准挑战。摘要必须保留架构决策、未解决的错误和实现细节，同时丢弃冗余的工具输出和已纳入后续步骤的中间推理。推荐的调整工作流程首先最大化召回率，从跟踪中捕获每条可能相关的信息，然后迭代通过删除多余的内容来提高精度。一个人不会从另一个方向开始。过早删除太多内容会产生智能体，它会丢失稍后需要的上下文，并且这种损失是不可恢复的。工具结果清除是最轻的压缩形式：一旦智能体对其起作用，完整的工具输出将被替换为压缩路径引用。这可以持续应用，现已成为 Anthropic 开发者平台 (Anthropic, 2025c) 上的产品级功能。

子智能体上下文隔离。当任务需要深入探索子主题时，探索本身会生成大量中间上下文，这些上下文在子任务中有用，但会污染协调器对整个任务的看法。 sub-智能体架构通过将子任务分配给专用智能体来解决这个问题，每个子任务都有一个新的上下文窗口，该窗口仅将其发现的精简的1,000到2,000个令牌摘要返回给协调器(Anthropic, 2025e)。详细的探索上下文在子智能体内保持隔离；协调器仅收到蒸馏结果。 Manus 团队(2025) 描述了此模式的改进。对于编排器仅需要输出的简单子任务，上下文根本不共享。对于需要共享状态的复杂子任务，编排器的完整上下文将传递给 sub-智能体。在 智能体之间共享上下文是昂贵的：它会产生更大的预填充并消除跨不同系统提示的 KVcache 重用。必须针对每种任务类型明确地权衡隔离成本和协调收益。

Harness级别的视角：检查点和恢复模式允许智能体在瞬态故障中幸存下来
在不丢失任务状态的情况下，生命周期挂钩可以在可配置的阈值下自动触发压缩或内存整合。这是正确的责任划分。上下文管理应该是基础设施的工作，而不是智能体的工作。当Harness自动处理压缩和合并时，模型可以专注于任务推理。

### 6.1 生命周期状态管理

生命周期状态管理涉及在轮次、会话、故障和切换中继续智能体运行所需的操作状态。这包括待处理的子任务、工具结果、中间工件、存储库更改、协调元数据、会话持久性和检查点恢复机制。这些机制使编排持久而不是短暂：智能体 可以继续之前的工作，而不是将每个步骤视为孤立的模型调用。

一个核心的权衡是无状态执行和有状态执行之间。无状态设计根据先前的交互历史重建执行，提高了可重复性和可审计性，但随着轨迹的成本变得昂贵
成长。  有状态设计将执行状态保存在文件、数据库、存储库、任务图或外部中
服务，改善连续性和恢复，但引入一致性和调试挑战(OpenAI, 2026b; Anthropic, 2026c)。因此，许多实用的 Harness 采用混合设计，将可重播的交互历史与持久的工件相结合。

### 6.2 单智能体内循环

单智能体内循环是智能体系统中的基本执行单元。单个智能体通过工具使用和反馈与环境重复交互，而多个智能体之间没有明确的协调。这种抽象是交互式编码、调试和问题解决的基础。

从概念上讲，单智能体循环遵循ReAct范式(Yao et al., 2023)，交错推理、行动和观察。正如 OpenAI 对 Codex 智能体循环 (OpenAI, 2026b) 的分析中所强调的那样，此类系统的行为不仅由模型决定，还由 Harness 决定，Harness 构造提示、调用工具、管理控制流并将工具输出反馈到后续步骤。

在这个级别上，主要的执行区别在于无状态重放和混合执行之间。  法典

### 6.3 多智能体编排模式

多智能体编排通过将多个智能体 与专门角色组合来扩展单智能体循环。多个智能体 Harness 可以将这些职责划分给智能体 或子代理，而不是要求一个智能体 单独计划、执行、检查和修改。这种结构支持任务分解、并行探索、批判、验证和聚合，使其比孤立循环更适合复杂任务。

这个设计空间中出现了几种重复出现的模式。分层编排使用更高级别的控制器将工作分配给 智能体或子代理并集成它们的输出。团队编排将一组专门的智能体公开为一个协调的团队，通常具有不同的命名职责。工作-

### 7.3 成本跟踪和优化

LLM 推理按代币定价，智能体 Harness 会放大成本风险，因为单个面向用户的任务可能会触发数十个 LLM 调用，每个调用都有自己的提示组装、工具结果注入和响应生成。可观察性要求是双重的：跟踪（知道代币花在哪里）和优化（为相同的结果花费更少的代币）。

在跟踪方面，TensorZero 提供了一个统一的 LLMOps 堆栈，将网关、可观测性、实验和优化捆绑到单个服务中，从而实现每次调用和每个任务的成本归因 TensorZero (2026)。 Helicone 采用极简方法作为插入式代理，无需更改代码即可增加成本和延迟监控，这对于那些想要成本可见性而不需要使用完整的可观察性平台 Helicone (2026) 的团队来说特别容易访问。

在优化方面，FrugalGPT Chen 等人。 (2023) 通过提出三种降低成本的策略提供了基础公式：即时适应、LLM 近似和 LLM 级联。它表明，自适应级联可以与 GPT-4 的性能相匹配，通过将更简单的查询路由到更便宜的模型，成本降低高达 98%。 GPTCache Bang (2023) 通过语义缓存层补充了这种方法，该层在到达 LLM 之前拦截重复或释义的查询，使用嵌入相似性来检索缓存的响应。路由和缓存一起成为生产成本优化堆栈中的重复构建块，并直接适用于 Harness 级检测，其中每个工具调用和上下文组装步骤都可以独立计量和路由。

QC-Opt Shekhar 等人。 (2024) 使用质量感知框架扩展了路由范例，该框架在预算限制下联合优化模型选择、令牌计数和延迟，使用 BertScore 预测器来估计输出质量，而无需调用目标模型。 TALE Han等人发现的代币弹性现象。 (2025) 揭示了一个重要的细微差别：对于思想链推理来说，将代币预算设置得太低可能会矛盾地增加代币消耗，因为模型超出了预算，并比适度预算的提示生成更多的代币。在基础设施层面，双池令牌预算路由 Liu 等人。 (2026b) 解决了服务端的成本问题，将同质 vLLM 队列划分为短上下文和长上下文池，并根据估计的代币预算路由请求。在 Azure LLM Inference 跟踪和 LMSYS-Chat-1M 上进行评估后，该方法将 GPU 小时数减少了 31-42%（预计在 A100 队列规模上每年可节省 286 万美元）。

对于 Harness 工程师来说，这意味着成本可观察性必须跨越多个层：API 级
令牌跟踪（Helicone、TensorZero）、应用程序级路由决策（FrugalGPT、QC-Opt）以及
基础设施级资源利用率（Dual-Pool、KV-cache占用）。 Anthropic 对 智能体式 编码评估中的基础设施噪音的研究 Anthropic (2026a) 提供了一个警示性补充：仅基础设施配置就可以将基准分数改变 6 个百分点 (p < 0.01)。这一发现表明，成本优化和评估保真度是紧密相连的，因为削减资源以节省成本可能会默默地降低 智能体性能，而这种性能在没有细粒度可观察性的情况下是不可见的。

### 8.1 Harness 作为任务到反馈生命周期的评估

Harness 感知评估应将报告的分数视为模型-Harness 对的属性，而不是单独模型的属性。这促使评估协议要么锁定不同模型的Harness，要么将Harness 配置作为显式实验因子(Bölük, 2026b) 进行改变。我们将Harness 评估组织为从任务到反馈的生命周期：一个结构化流程，从定义要求智能体 执行的操作开始，到将评估结果反馈到Harness 改进中结束。图 12 总结了这个从任务到反馈的五个阶段的生命周期。该生命周期建立在传统 LLM 评估和 智能体评估之间的关键区别之上。传统的 LLM 评估主要对固定输入的输出进行评分，而Harness 评估则衡量执行事件：任务基于环境，智能体 随着时间的推移与工具和状态交互，捕获痕迹，评估者判断最终结果和达到它所采取的路径(Kapoor et al., 2025; Anthropic, 2026b; LangChain, 2025b)。

此生命周期的一个核心动机是评估基础设施噪音可能会伪装成模型失败：失败的运行可能不仅反映了模型限制，还反映了损坏的工具、过时的上下文、不可重置的沙箱、不稳定的测试、基准模糊性或不稳定的判断(Kapoor et al., 2025; Hu et al., 2025a; Jimenez et al., 2024)。因此，评估应该将智能体行为转化为结构化判断、失败归因和回归反馈，而不仅仅是报告最终分数。

• 任务和基准基础定义了评估内容、环境、工具、约束条件和成功标准（§8.2）。

• 执行前准备验证检查沙箱、依赖项、工具、上下文状态、权限策略、预算和评分器是否在智能体 运行之前正确初始化（第8.3 节）。

• 受控执行和跟踪捕获在可重现的条件下运行智能体，同时记录模型输出、工具调用、状态更改、错误、重试、成本和延迟（第8.4 节）。

• 多级判断和故障归因在结果、轨迹和评估者级别评估运行，然后在Harness 堆栈中定位可能的故障源（第8.5 节）。

• 持续回归和部署反馈将评估结果转换为回归测试、监控信号和工程反馈，以供后续Harness 修订（§8.6）。

### 8.2 第一阶段：任务和基准基础

第一阶段询问正在评估什么。对于LLM 智能体，任务不仅仅是自然语言提示，而是由环境状态、可用工具、允许的操作、约束定义的嵌入问题，

<!-- page 34 -->

终止条件和成功标准。因此，任务落地是Harness评估的基础：如果没有明确的环境和成功条件，后面的分数就无法可靠地解释。

### 8.2.1 软件工程和终端任务

软件工程基准测试是 智能体评估最成熟的形式之一，因为它们将实际任务与可执行结果验证结合起来。 SWE-bench 将每个任务基于真实的 GitHub 问题和存储库快照，然后通过运行测试 (Jimenez et al., 2024) 来评估生成的补丁是否解决了问题。 Terminal-Bench 将这一想法扩展到命令行工作流程，其中 智能体通过编辑文件、运行命令、设置依赖项和解释故障 (Merrill et al., 2026) 与终端环境进行交互。这一类别的关键见解是，强大的结果验证者需要强大的任务基础。仅当精确指定存储库状态、依赖性和预期成功标准时，测试才能充当可靠的评估者。否则，失败的测试可能反映无效的补丁，但也可能反映不一致的环境或未指定的基准实例。

### 8.2.2 Web、浏览器和计算机使用任务

Web 和计算机使用基准测试交互式环境中的地面任务，其中成功由浏览器、桌面或应用程序状态变化来定义。 WebArena 为用户引入了真实的网络环境
调皮的智能体(Zhou et  al., 2024)； VisualWebArena 添加多模式 Web 任务(Koh et  al., 2024)；
BrowserGym提供通用浏览器-智能体研究基质(Chezelles et al., 2024)； WorkArena 专注于基于 ServiceNow 的知识工作(Drouin et al., 2024)。 OSWorld 进一步将评估范围扩大到真实的计算机环境，包括桌面应用程序、文件和多应用程序工作流程(Xie et al., 2024)。这些基准测试表明，浏览器和计算机使用评估同时是一个评估问题和一个环境设计问题。基准测试必须提供一个现实的界面，隔离任务状态，公开适当的观察结果，并定义可衡量的成功谓词。这在执行环境层和评估层之间建立了直接的桥梁。

### 8.2.3 跨域和企业工作流任务

第三组基准测试跨异构工具、环境和工作流程更广泛的 智能体功能。 AgentBench 跨操作系统、数据库、网页浏览和游戏(Liu et al., 2023a) 评估LLM 智能体； GAIA 针对需要推理、工具使用和多模式信息访问的一般助理任务(Mialon et al., 2023)； TheAgentCompany 模拟工作场所式的数字劳动力(Xu et al., 2024)。 WorkArena 和 WorkArena++ 进一步强调企业工作流程，其中成功取决于导航复杂的软件，而不是回答孤立的问题(Drouin et al., 2024; Boisvert et al., 2024)。该基准测试系列的主要贡献是覆盖范围：它测试Harness是否可以跨任务类型、工具接口、状态表示、工作流程和成功条件进行泛化。

### 8.3 第 2 阶段：执行前准备情况验证

第二阶段询问评估是否可以公平且可重复地进行。这个阶段在基准排行榜中通常是不可见的，但它是Harness评估的核心。在 智能体开始之前，Harness 必须验证环境、工具、上下文状态、权限边界、预算约束和评估器是否已正确初始化。如果没有此准备层，下游故障将难以归因：运行失败可能是由 智能体引起的，但也可能是由评估设置引起的。

### 8.3.1 环境和沙箱准备情况

环境准备情况检查沙箱、存储库、浏览器、终端或虚拟机是否从已知基线启动。在软件工程中，这包括存储库快照、依赖项、任务设置和测试可执行性；在浏览器和计算机使用设置中，它包括网站重置、浏览器配置文件、桌面状态和服务可用性。

有几个系统使这个问题变得明确。 SWE-bench 依赖于可执行存储库状态和基于测试的验证(Jimenez et al., 2024)； Terminal-Bench 将终端任务与受控环境打包在一起

验证逻辑(Merrill et al., 2026)； OSWorld 使用真实的计算机环境和基于执行的评估脚本(Xie et al., 2024)。 Repo2Run 通过自动化存储库 (Hu et al., 2025a) 的可执行 Docker 环境来实现依赖关系和环境设置，而 HAL 则强调标准化、
智能体 (Kapoor et al., 2025) 的成本感知评估基础设施。  这些系统共同表明
环境重置和依赖性验证是测量工具的一部分。

### 8.3.2 工具、上下文和权限准备情况

准备情况验证是跨层的。  工具准备情况检查 API、MCP 服务器、浏览器控件、
shell 命令和文件操作可用且描述一致。  上下文准备情况检查
历史记录、内存存储、暂存器和检索到的文档是否被重置或有意初始化。权限准备情况检查文件访问、凭据、网络访问和批准门是否符合基准规范。

此阶段将评估与工具、上下文和治理层连接起来。如果工具描述在运行过程中发生变化，基准测试将不再测量相同的操作空间。如果内存或上下文未重置，智能体

### 8.3.3 评估员和分级员准备情况

评估器本身也必须在执行之前进行验证。应检查确定性分级器是否不稳定、缺少依赖性以及与环境状态的兼容性；法学硕士作为法官的提示、评分标准和法官模型应该进行版本控制；应提前定义人工审核协议。

在 SWE-bench 等基准测试中，可执行测试提供可扩展的结果验证，但分数有效性仍然取决于正确的环境设置、任务规范和测试可靠性(Jimenez et al., 2024)。对于开放式任务，LLM-as-Judge系统需要校准、偏差缓解和一致性检查，而不是盲目地重复使用强大的模型作为评分者(Zheng et al., 2023; Gu et al., 2024)。因此，评估者的准备就绪是进行有意义的失败归因的先决条件。

### 8.4 第三阶段：受控执行和跟踪捕获

第三阶段询问执行过程中发生了什么。在传统的法学硕士评估中，证据可能由单个输入-输出对组成。在智能体评估中，跟踪是多步轨迹：模型观察状态、推理、调用工具、接收工具输出、修改环境、处理错误并最终终止。因此，为了将基准运行转化为可诊断的证据，受控执行和跟踪捕获是必要的。

### 8.4.1 受控部署和可重复执行

部署是智能体评估的基本单位。它包括任务、模型配置、Harness配置、操作序列、中间观察、最终状态和评分结果。受控部署修复了意外变化的关键来源，包括环境状态、工具可用性、超时、预算、权限策略和评估器版本。当不确定性仍然存在时，重复推出可以暴露差异，而不是将其隐藏在单个分数后面。

几个系统说明了对可重复执行的需求。 SWE-agent 将智能体 暴露给存储库导航、文件编辑和测试执行，作为交互循环(Yang et al., 2024) 的一部分。 OpenHands 结合了代码编辑、命令行执行、浏览器交互、沙盒执行和基准测试集成(Wang et al., 2025b)。 SWE-ReX 对本地和远程沙盒 shell 环境进行抽象

<!-- page 36 -->

这样执行后端就可以在不更改智能体逻辑(SWE-agent Team, 2024)的情况下进行更改。 LangChain进一步将深度智能体评估分解为单步验证、全轮评估和多轮模拟(LangChain, 2025b)。这些系统共同表明，再现性需要重播完整的模型-工具-环境交互，而不仅仅是重新运行模型调用。

### 8.4.2 跟踪本机评估

跟踪捕获将二进制评分转换为因果诊断。本地跟踪 Harness 应记录模型输出、工具调用、工具结果、环境状态更改、上下文快照、错误、重试、恢复操作、令牌使用、延迟和成本。这些痕迹区分了表面上类似的故障：一个 智能体可能永远找不到相关文件，而另一个可能找到它但产生无效补丁。它们还可能揭示不良的成功，例如基准利用、过多的工具调用或权限违规。

HAL 将日志和跟踪视为标准化智能体 评估的一流工件，使用大规模智能体 日志来检查最终分数可能隐藏(Kapoor et al., 2025) 的行为。 R2E-Gym 同样将可执行轨迹和混合验证器链接到测试时评估和训练时反馈(Jain et al., 2025)。因此，对于 Harness 工程，跟踪不是辅助调试工件；它们是主要的评估数据。

### 8.4.3 成本、延迟和资源跟踪

智能体 评估不仅应该报告质量，还应该报告实现质量的成本。长期智能体可以通过花费更多代币、调用更多工具、更频繁地重试或使用更强大的模型来提高成功率。因此，Harness 级评估应跟踪令牌使用情况、模型成本、​​挂钟延迟、工具调用计数、重试计数和资源消耗。

评估不应仅按成功率对 智能体进行排名，而应揭示成功-成本-延迟边界。 HAL 明确将智能体 评估框架为标准化和成本意识(Kapoor et al., 2025)，而LangChain 的深度智能体 评估指南强调可重复环境和多粒度评估，以比较成本质量权衡(LangChain, 2025b)。这对于已部署的 智能体服务尤其重要，其中优化的单位是预算和延迟限制下的重复工作负载。

### 8.5 第四阶段：多层次判断与失败归因

在受控执行和跟踪捕获之后，中心问题是如何判断运行，如果运行失败，如何定位故障。我们将第四阶段定义为多层次判断和失败归因的结合过程。多层次判断从三个层面对运行进行评估：最终结果是否正确、轨迹是否高效且符合政策、评估者本身是否可靠。然后，故障归因使用这些信号来诊断故障最有可能源自的位置，例如模型、工具接口、上下文管理器、执行环境、编排循环、基准规范或评估器。这个阶段是Harness评估与普通基准评估的区别：目标不仅仅是打分，而是将执行证据转化为可操作的工程诊断。

### 8.5.1 结果层面评估

结果级评估衡量最终任务目标是否实现。在软件工程中，这通常通过单元测试、回归测试或特定问题的验证来实现，如 SWEbench (Jimenez et al., 2024) 中。在终端任务中，结果评估可能取决于最终文件、命令输出或特定于任务的检查器(Merrill et al., 2026)。在浏览器和企业工作流基准测试中，它通常取决于最终环境状态，例如是否提交表单、更新票证或更改配置(Zhou et al., 2024; Drouin et al., 2024)。在 GAIA 等一般助理基准测试中，结果可能是答案字符串或结构化响应(Mialon et al., 2023)。

结果级评估的优势在于可扩展性和可解释性：它提供了明确的任务级
成功信号并支持排行榜比较。它的局限性是压缩。完整的执行事件被简化为一个值，隐藏智能体是否稳健、安全或高效地成功。结果评估

<!-- page 37 -->

因此是必要的但还不够。  它回答任务是否完成，但不回答任务是否完成
Harness 产生了值得信赖的执行。

### 8.5.2 轨迹级评估

轨迹级评估衡量 智能体所采取路径的质量。它询问 智能体是否选择了适当的工具、明智地安排了操作、避免了冗余调用、尊重权限边界、从错误中恢复以及随着时间的推移保持上下文一致性。此级别是 智能体评估与传统输出评估最明显的分歧：最终答案可能是正确的，但轨迹仍然不可接受，例如，如果 智能体浪费过多的令牌、重复调用不相关的工具、违反权限或依赖于脆弱的基准特定行为。

ReAct 通过将智能体 行为表示为交错的推理、行动和观察步骤(Yao et al., 2023)，为轨迹级分析提供了概念基础。现代 智能体基准测试将此视图扩展到更丰富的环境中，其中每个操作都会更改未来步骤可用的状态。 WebArena、WorkArena、OSWorld 和 Terminal-Bench 都生成轨迹，其中间操作是有意义的评估对象，而不是隐藏的实现细节(Zhou et al., 2024; Drouin et al., 2024; Xie et al., 2024; Merrill et al., 2026)。 LangChain的评估模式同样强调单步和全轮评估，因为它们允许开发人员在整个运行成功或失败之前检查决策质量(LangChain, 2025b)。

轨迹级判断对于Harness工程特别有价值，因为它提供了从故障到修复的途径。如果 智能体由于选择了错误的工具而失败，则工具界面可能需要改进。
如果它忘记了先前的约束，则上下文层可能需要调整。  如果它循环而没有恢复，
编排层可能需要生命周期控制。因此，轨迹评估将基准结果转化为特定于层的工程反馈。

### 8.5.3 评估者级评估

评估者层面的评估询问的是评估者本身是否可信。确定性验证器（例如单元测试或基于状态的检查器）可重复、便宜且易于跨系统比较，但它们范围狭窄且难以针对开放式任务进行设计。法学硕士作为法官系统对于自然语言输出和轨迹评估来说是灵活的，但它们引入了偏差、不确定性和额外成本。对于不明确或安全关键的情况，人工审核仍然很重要，但它们无法扩展到持续评估。

G-Eval 表明，基于 LLM 的评估者可以更好地与人类对 NLG 评估任务的判断保持一致，同时也强调了对 LLM 生成的文本 (Liu et al., 2023b) 的偏见。 MT-Bench 和 Chatbot Arena 系统地研究了 LLM-as-Judge 并发现了立场偏差、冗长偏差、自我增强偏差和有限推理能力(Zheng et al., 2023)。最近的调查进一步表明，可靠的法学硕士作为法官系统需要标准化、减少偏见、一致性检查和元评估(Gu et al., 2024)。

对于智能体 Harness，评估者级别的评估不是可选的。如果评分者不稳定，测试不确定，或者 LLM 法官有偏见，那么评估噪声可能会被错误地归因于 智能体或模型。因此，一个强大的Harness 应该更喜欢分层评分器：对客观状态变化进行确定性检查，对语义或轨迹级评估进行LLM法官，并对模糊或高风险案例进行人工审核。评估器应该被视为被测试的组件，而不是系统之外的预言机。

### 8.5.4 跨Harness层的故障归因

失败归因确定了判断运行后应修复的问题。失败的推出可能源于模型推理、工具界面设计、陈旧或压缩的上下文、沙箱不稳定、编排循环、基准模糊性或评估器不可靠性(Kapoor et al., 2025; Hu et al., 2025a; Jimenez et al., 2024; LangChain, 2025b)。结果层面的信号表明目标是否实现；轨迹级信号揭示行动序列是否连贯、高效且符合政策；评估者级别

<!-- page 38 -->

信号表明分数本身是否值得信赖。这些信号共同支持诊断，而不仅仅是评分(Kapoor et al., 2025; LangChain, 2025b)。

在实践中，归因很少是单标签分类问题。模糊的任务规范可能会导致智能体选择错误的工具；过度压缩的上下文可能会导致它忘记约束；沙箱依赖性问题可能会触发恢复行为，从而增加成本；否则，一个反复无常的法官可能会掩盖有效的解决方案。因此，故障归因应被理解为整个跟踪的诊断过程，而不是附加到最终分数的事后标签。因此，第 4 阶段的输出不仅是判断，而且是第 5 阶段可以反馈到回归测试和 Harness 改进的结构化诊断。

### 8.6 第 5 阶段：持续回归和部署反馈

最后阶段询问评估结果如何推动下一个Harness修订。 智能体 Harness不断发展：提示、工具模式、MCP服务器、上下文策略、沙箱图像、编排循环、治理规则和判断提示都可能随着时间的推移而改变。持续评估将基准测试结果和生产故障转化为 Harness 本身的回归系统。

### 8.6.1 Harness 变化的回归评估

回归评估应由Harness 更改以及模型更改触发。对工具描述、上下文压缩策略、沙箱图像、权限规则或判断提示的更改可能会改变智能体行为或评估分数。由于Harness组件相互作用，局部改进可能会导致全局回归。

实用的模式是维护一个分层的评估套件：针对工具模式和de-的类似单元的测试
术语验证器、本地决策的单步测试、端到端完成的全面部署测试以及长期一致性的多轮模拟。 LangChain的深度智能体评估指南反映了这种分层视图(LangChain, 2025b)，而Anthropic框架则评估为具有明确分级逻辑的自动化测试，可以在开发过程中运行(Anthropic, 2026b)。

### 8.6.2 评估框架和 LLMOps 集成

通用评估框架为持续测试提供可重用的基础设施。  提示foo支持-
ports提示和LLM申请回归测试； DeepEval 提供类似单元测试的抽象； RAGAS专注于检索增强生成评估； lm-evaluation-Harness 支持标准化语言模型评估(promptfoo contributors, 2026; Confident AI, 2026; Es et al., 2024; Biderman et al., 2024; Gao et al., 2021)。尽管这些工具并非都是为长期运行智能体而设计的，但它们为面向回归的Harness评估提供了构建块。

评估还应该与可观察性联系起来。  生产痕迹可以成为回归测试，并且
评估失败可以成为可观察性信号。  此链接关闭了监视内容之间的循环
发生并构建重现和诊断它的受控测试。

### 8.6.3 基于验证者和训练时的评估

一个新的方向不仅使用评估器作为离线指标，而且还作为训练信号和优化目标。基于验证者的环境和 RL 风格 智能体健身房（包括 R2E-Gym 和验证者）使用环境反馈作为奖励或验证信号，以改进 智能体政策和脚手架 (Jain et al., 2025; Prime Intellect, 2026)。这将评估从事后测量步骤转变为 智能体培训、测试时间适应和支架选择的主动组成部分。

Meta-Harness 通过将Harness 设计本身视为自动搜索(Lee et al., 2026) 的对象来扩展此方向。该视图不是只评估哪个模型在固定Harness下表现最好，而是询问哪个Harness结构、提示策略、工具界面或控制循环产生更可靠的智能体行为。在这个框架中，评估并不是流程的终点；而是流程的终点。这是允许Harness 改进的信号。

### 9.1 权限模型和身份管理

对于任何 智能体 Harness 来说，第一个治理问题是访问控制：智能体 可以访问哪些工具、文件、网络端点和系统资源？ 智能体工作负载使这比传统的 RBAC 或 ABAC 更难，因为所需的工具集通常取决于部署时未知的自然语言任务。因此，文献沿着粒度轴移动：部署时固定的静态边界、每个工具调用评估的上下文策略以及本地 Harness 之外交互的跨边界权限。

<!-- page 40 -->

静态权限边界。  生产编码智能体通常预先定义一个权限范围和
需要升级其外部的操作。 Codex (OpenAI, 2025) 在文件系统和网络访问受限的沙箱中运行 shell 命令，而 Gemini CLI (Mullen & Salva, 2025) 将工作区范围的文件访问与命令允许和拒绝列表结合起来。这些边界很容易检查，但它们无法表达特定于任务的意图。

超越固定规则需要表达依赖于每个工具调用的运行时上下文的约束。

上下文相关的权限控制。  Progent (Shi et al., 2025a) 引入了一种特定于领域的语言
(DSL)，其谓词范围涵盖工具名称、参数和环境状态。运行时在每次调用之前评估这些谓词，允许最小权限策略根据角色、用户或会话而变化。 Conseca (Tsai & Bagdasarian, 2025) 通过从受信任的上下文生成特定于任务的策略并使用单独的确定性检查器强制执行它，进一步推动了这一想法。政策生成和执行之间的分离很重要：生成组件可以适应任务，而执行者仍然可审计。

上述方法在单个 智能体边界内运行。多智能体设置在访问控制和智能体 身份方面带来了额外的挑战。

身份管理和智能体间访问控制。  在授予访问权限之前，系统必须建立
lish 谁在请求它。南等人。 (2025) 认为 智能体需要经过身份验证的委派：一个扩展 OAuth 2.0 和 OpenID Connect 的框架，其中包括用户 ID 令牌、智能体 ID 令牌和作用域委派令牌，这些令牌将用户意图绑定到 智能体特定权限。该代币链将创建从人类主体到智能体行动的可验证的责任追踪。 SAGA (Syros et al., 2025) 通过提供者介导的架构建立在多智能体设置的原则之上。 智能体使用加密一次性密钥注册，短期访问令牌在 智能体之间强制执行用户定义的联系策略。 IsolateGPT (Wu et al., 2025) 采用架构方法，采用中心辐射模型，其中每个
第三方应用程序在独立的分支中执行，具有自己的 LLM、内存和工具访问权限。  间
分支通信通过受信任的中心，因此跨应用程序数据交换成为明确的治理决策，而不是共享上下文窗口的偶然副作用。

凭证管理。 智能体例行处理 API 密钥、会话令牌和一次性密码。将这些凭证暴露给 LLM 环境会产生渗透风险。 Skyvern (Skyvern-AI, 2025) 说明了一种常见模式：将机密存储在保管库中，仅向 LLM 公开占位符，并且仅在执行经过身份验证的操作的自动化层替换原始值。未解决的问题是长期会话的秘密生命周期管理，其中令牌可能会过期或在轨迹中被撤销，并且更新的凭证必须仍然保留在模型上下文之外。

Web级别的权限协调。  上述机制在单个Harness的信任内运行
边界并管理一个部署。跨组织交互（智能体 遍历第三方网站或调用其他方拥有的服务）需要进行协调，而单个 Harness 无法单独执行。马罗等人。 (2025) 提出智能体-permissions.json，一个类似于robots.txt 的轻量级清单文件。网站所有者可以声明 智能体可以使用哪些 UI 元素、速率限制、并发约束以及需要人工确认的操作。此类清单不能解决身份验证问题，但它们显示了治理如何开始跨越组织边界。

开放挑战。  设计既具有表现力又可用的权限模型是一个未解决的问题
莱姆。过于严格的政策会降低智能体的实用性；过于宽容就会否定治理的目的。社区尚未达成可跨 Harness 实现移植的标准权限规范语言。关于操作是否应归因于操作员、智能体 实例或临时任务身份(South et al., 2025)，也仍然存在悬而未决的问题。

预执行挂钩：输入护栏。输入护栏在数据到达 LLM 之前验证数据。 PromptShield (Jacob et al., 2024) 和 DataSentinel (Liu et al., 2025) 训练专用分类器来检测用户输入和检索内容中的提示注入有效负载。基于规则的检测器严格但脆弱。基于模型的检测器泛化能力更好，但速度较慢并且容易受到自适应攻击(Andriushchenko et al., 2024; Nasr et al., 2025)。

下一个执行点发生在 LLM 提议的操作和 Harness 的工具执行之间。

预调用挂钩：输出护栏和操作验证。  在执行工具调用之前，输出
护栏检查法学硕士提出的行动。 ShieldAgent (Chen et al., 2025b) 正式规定安全约束
作为可验证的谓词并根据它们检查每个操作。  在多 智能体系统中，ControlValve (Jha
et al., 2025）限制智能体之间的控制流图并防止未经授权的智能体转换，解决控制流劫持攻击，其中一个智能体将执行重定向到另一个。

工具执行后，在重新进入 LLM 上下文之前还必须检查返回的数据。

执行后挂钩：信息流控制和污点跟踪。  CaMeL（Debenedetti 等人，
2025）实现基于能力的信息流控制。每个数据值都带有跟踪其来源的元数据标签，区分受信任的用户输入和不受信任的 Web 检索。自定义解释器强制不受信任的数据不能影响控制流决策。 CaMeL 将受信任上下文的规划与不受信任内容的处理分开，以一定的灵活性换取 AgentDojo (Debenedetti et al., 2024) 上更强大的控制/数据流边界。

互补的执行机制将人类用户置于循环之中。

人机交互挂钩。  几种广泛使用的编码智能体，包括Codex、Gemini CLI、Cursor、
和 OpenHands (Wang et al., 2025a)，在用户明确批准的情况下进行破坏性或超出范围的操作。三个设计维度塑造了这些钩子：验证范围（哪些操作需要批准）、警报丰富度（用户看到多少上下文）和重复策略（允许一次与始终允许）。费尔特等人。 (2012) 表明，只有 17% 的 Android 用户在应用安装过程中关注权限对话框，只有 3% 的用户正确回答了有关他们所授予权限的理解问题； 智能体批准对话框可能面临类似的习惯和理解风险。频繁的请求会导致用户本能地批准危险的操作，而不频繁的请求会留下覆盖范围的空白。

一些系统将钩子视为可编程的执行基础，而不是孤立的分类器。 AgentSpec (Wang et al., 2026) 定义了带有触发器、谓词以及计划、工具调用和响应阶段的执行操作的规则。 AgentDoG (Liu et al., 2026a) 从单动作分类转变为轨迹级诊断，按来源、故障模式和后果组织风险。

开放挑战。  在当前的系统中，hook API 仍然是异构的。我们不知道广泛
采用的标准为第三方治理模块定义了通用钩子接口。钩子还会引入延迟，并且堆叠钩子之间的交互效果在很大程度上仍未被研究。一种可能的故障模式是上游消毒器改变了下游检测器所依赖的信号，使组合管道比单独考虑的任何一个组件都弱。

### 9.3 组件强化

生命周期挂钩在 Harness 级别强制实施治理。组件强化增强了各个 智能体组件，特别是 LLM 及其调用的工具，以应对其特定的漏洞。 Harness 受益于强化组件，因为能够首先触发治理挂钩的恶意输入较少。

模型强化。提示注入的根本原因是LLM以同等优先级对待所有输入文本，这使得它们无法区分系统指令和不受信任的数据。华莱士等人。 (2024) 提出了一种指令层次结构，用于训练模型将特权指令（系统提示）优先于低特权指令（用户消息、工具输出）。当较低级别的指令与较高级别的约束一致时，模型会学习与它们保持一致，并在它们发生冲突时忽略或拒绝它们。 SecAlign (Chen et al., 2025a) 制定了与提示注入输入、期望响应和不良响应的偏好优化相同的防御目标。

这些模型级防御补充了Harness级挂钩。强化模型在推理时拒绝更多注入，从而减少下游护栏的负载。然而，仅靠模型强化是不够的：它解决了 Kim 等人的分类中遵循的错误指令，但不能防止不受约束的数据流或未经授权的操作，这需要系统级强制(Kim et al., 2026; Wei et al., 2026)。

基于分类器的运行时强化。通过部署在运行时筛选输入和输出的小型辅助分类器，互补的工作线在不修改 智能体LLM 本身的情况下强化了模型边界。 Llama Guard (Inan et al., 2023) 微调了一个单独的模型，以根据可配置的安全分类法对用户提示和助理响应进行分类，返回Harness 可以路由的每个类别标签以允许、阻止或升级决策。此类分类器提供了训练时强化所缺乏的两个实用属性：可以在不重新训练 智能体模型的情况下编辑分类法，并且可以在异构 LLM 后端中重复使用相同的检测器。成本是第 9.2 节中讨论的延迟预算：每个附加分类器都会在每个工具使用周期添加一次前向传递。

模型和分类器防御强化了 LLM 边界；工具边界需要有自己的协议级处理。

工具强化和 MCP 安全。  模型上下文协议 (MCP) (Anthropic, 2024c) 具有
作为 智能体和工具之间的通用接口出现，但其初始规范缺乏本机安全原语。这个差距引起了攻击和防御，以及官方规范的回应。 Radosevich 和 Halloran (2025) 证明了广泛使用的商业法学硕士可能会被强迫使用 MCP 工具来执行恶意代码、建立远程访问和窃取凭证。他们的McpSafetyScanner使用三智能体架构（黑客、审计员、主管）来系统地发现此类漏洞。 Trail of Bits(Trail of Bits, 2025)表明，MCP服务器可以在用户调用工具之前通过有毒的工具描述来攻击客户端，从而影响LLM在注册时的行为。

在防御方面，ETDI (Bhatt et al., 2025) 使用加密签名和版本化工具定义扩展了 MCP，确保对工具代码、架构或权限的任何修改都需要新签名的
版本。  这可以防止拉动攻击，即先前批准的工具会默默更新以执行
恶意行为。

协议级强化。  除了单个组件之外，SAFEFLOW (Li et al., 2025) 还引入了
多智能体系统的协议级强制执行。  它将细粒度的信息流控制与
事务执行语义，因此违规工具调用或 智能体间消息可以回滚，而不是通过共享状态传播。

供应链风险。 智能体治理还必须解决智能体 调用的工具和包的问题。斯普拉克伦等人。 (2025) 分析了 16 个法学硕士的 576,000 个代码样本，发现开源模型以 21.7% 的比率幻觉不存在的包名称。攻击者可以在公共存储库上注册这些幻觉名称并注入恶意代码，这种技术称为“slopsquatting”。 ETDI 的加密签名解决了 MCP 工具的部分问题，但包管理器和检索源仍然位于大多数 智能体治理堆栈之外。

开放挑战。  模型强化和工具强化解决了互补的威胁面，但没有
统一的框架将它们连接起来。强化的模型仍可能调用受损的工具，并且签名的工具定义并不能防止越狱模型的误用。将这些防御措施组合成一个具有可量化残余风险的连贯堆栈仍然是一个悬而未决的问题。随着 MCP 采用的不断增长，其安全扩展自然成为标准化目标，但互操作性和认证在很大程度上仍悬而未决。

### 9.4 声明性宪法

将治理逻辑直接嵌入到应用程序代码中会使策略不透明、难以审计且难以更新。越来越多的 Harness 将治理规则外部化为声明性配置文件，最常见的是 YAML 格式。这些文件充当智能体的机器可读构成。

训练时宪法：Anthropic 的宪法人工智能。  人类(Anthropic, 2026a) pub-
制定了一个宪法，分为四层优先级：安全（保留人类监督）、道德（诚实、避免伤害）、合规（Anthropic 准则）和帮助（用户请求）。该宪法区分了硬约束，例如绝对禁止化学、生物、放射性和核（CBRN）援助，与运营商可以在规定范围内调整的软编码默认值。在智能体式 设置中，此结构建议了一个主要层次结构，其中 Anthropic 的策略覆盖操作员配置，而操作员配置又覆盖最终用户的首选项。

训练时宪法通过在对齐和训练后塑造模型的行为来运作。它们在训练时塑造模型行为，但很难在部署后修改，并且无法由 Harness 独立审核。一种独特的方法将治理规则外部化到 Harness 可以读取、验证和强制执行的部署时配置中。

部署时宪法：基于 YAML 的治理。  AutoHarness 存储库 (AIMing
Lab，2026）将治理规则编码在 YAML 文件中，该文件指定管道模式（核心、标准或增强）、风险分类模式、允许和拒绝的工具模式、代币预算限制和审计
记录目的地。  这种声明式风格将治理意图与执行逻辑分开。  非开发者
安全团队和合规官员等利益相关者可以在不接触的情况下审查和修改策略
智能体 代码。  与训练时的构成不同，YAML 文件可以更新、版本化和差异化
标准工具，无需重新训练或重新部署模型即可实现策略更新。

这两层在操作上有所不同。训练时的宪法修改成本很高，并且必须通过行为探测来推断，而部署时的 YAML 是直接可读和可差异审查的。训练时间对齐以概率方式塑造默认行为，并且可以被对抗性提示(Andriushchenko et al., 2024)覆盖；部署时规则改为以 Harness 检查的形式执行。

当策略简化为文字触发器时，基于模式的 YAML 规则就足够了，但实际部署通常需要运行时状态上的条件逻辑。当前的 YAML 模式没有标准化特权谓词
升级、基于主机的出口、会话级速率限制或未来的操作。  在自由格式 YAML 和
硬编码规则存在于第三个设计点：结构化策略 DSL。

可编程策略语言。  Progent 的策略语言 (Shi et al., 2025a) 支持布尔 pred-
icates、量词和环境引用。它提供了形式表达，同时保持可读性。正式 LLM (Li et al., 2024) 进一步将计划约束编码为下推自动机。这种形式可以表达顺序排序约束、禁止的工具组合以及特定步骤所需的批准门。 VeriSafeAgent (Lee et al., 2025) 通过 UI 状态转换将用户意图形式化为 DSL，

<!-- page 44 -->

在执行之前验证建议的 GUI 操作是否与用户的任务相符。 YAML 章程承认含糊不清；正式的规范需要专业知识。

开放挑战。  智能体宪法不存在广泛采用的标准模式。每个Harness都倾向于
定义自己的 YAML 结构，使策略不可移植。在当前的生态系统中，用于验证宪法内部一致（例如，没有矛盾的允许/拒绝规则）和完整（例如，没有未处理的工具类别）的工具似乎受到限制。训练时间和部署时间构成之间的相互作用尤其未被充分探索。 YAML 拒绝规则是否可以可靠地覆盖 RLHF 强化的行为，以及在什么条件下两层发生冲突，仍然是一个具有实际安全影响的悬而未决的问题。

### 9.5 审计基础设施

治理需要问责制。  审计基础设施记录 智能体做了什么、为什么这样做以及
治理政策是否得到尊重。这些记录支持事后调查、监管合规性和持续的政策完善。

结构化审计跟踪。  AutoHarness (AIMing Lab, 2026) 发出每个工具调用的 JSONL 记录，其中 -
包括工具名称、参数、风险分类、权限决策、执行结果、令牌成本和挂钟延迟。这些结构化日志支持实时仪表板和离线取证分析。 SAGA (Syros et al., 2025) 通过记录智能体 之间交换的加密令牌，将审计跟踪扩展到多智能体交互，从而实现跨信任边界的来源跟踪。在这些系统中，可重播的审计记录至少需要跟踪标识符、主体身份、工具调用、策略决策和版本、执行结果、资源成本以及相关输入和输出的完整性哈希。大多数当前系统仅记录这些字段的子集，以及很少的符号或哈希记录，从而使审计跟踪容易受到受损智能体进程的日志篡改。

除了记录之外，检测长期运行的 智能体中的治理违规行为还需要自动分析。

异常检测：每个动作与轨迹级别。  检测机制沿着格兰-
他们评估证据的清晰度。每个操作检测器根据学习的或指定的风险概念对每个工具调用进行分类：输入和输出护栏（第 9.2 节）在此机制中运行，AgentMonitor (Naihin et al., 2023) 提供轻量级日志和标志实现。这种机制成本低廉且易于审核，但它无法识别分布在许多单独的良性操作中的攻击，例如每分钟读取一个文件的缓慢渗漏。轨迹级检测器评估序列
共同行动。  AgentAuditor (Luo et al., 2025) 将行为模式匹配与基于 LLM 的方法相结合
对完整痕迹进行推理。 SentinelAgent (He et al., 2025) 将多智能体通信建模为时间图并标记异常交互模式。轨迹级检测捕获多步攻击，但代价是更高的延迟和不太本地化的操作触发警报的概念，使实时干预和事后解释变得复杂。在这里调查的系统中，每次操作检查通常是内联部署的，而轨迹级分析则在审核日志上异步运行。

除了安全性之外，长期运行的 智能体还需要成本和资源治理。

通过其宪法以声明方式制定预算。  具有成本意识的治理在多智能体中尤其重要
扇出模式可以成倍增加 API 调用的架构。

分层治理管道。  一种新兴的设计模式集成了前面的机制（per-

开放挑战。  审计日志在长期运行的 智能体中快速积累，引发了存储方面的挑战，

### 9.6 将治理置于 智能体安全环境中

治理机制应对具体的威胁形势。来自开放 智能体到智能体 平台的最新证据(Zhang et al., 2026b; Kim et al., 2026; Chen et al., 2026; Wei et al., 2026) 进一步表明，智能体 安全性不仅限于提示注入或单智能体越狱：社会工程、协调智能体 集群、凭证泄露和平台级参与放大可以共同塑造威胁格局。金等人。 (2026) 调查了 智能体堆栈中的 128 篇论文并列出了 51 种攻击方法和 60 种防御方法。陈等人。 (2026) 从软件工程的角度对此进行了补充，通过六维分类法综合了 50 篇论文，并提出了安全构建 智能体平台的参考原则。我们使用这些框架将治理机制与特定的安全风险联系起来并确定覆盖范围差距。

## 10 个跨领域问题

本节中收集的问题跨越多个 ETCLOVG 层，并通过显示特定于层的推理在何处发生故障来开发权利要求 1（智能体 Harness 作为独立系统层，§1）。各层章节隔离了执行、工具、上下文、生命周期、可观察性、评估和治理，以便可以精确地描述每个设计表面。然而，生产Harness最常在这些表面之间的界面处失败。

层间交互模式。这七层并不是独立的。执行环境（E）限制了哪些生命周期和编排策略（L）是实用的；情境管理 (C) 影响评估的可重复性 (V)；治理 (G) 施加跨越所有其他层的身份、权限和审计约束。因此，Harness 设计应被视为依赖结构，而不是可分离组件的清单。

成本-质量-速度三难困境。更强的评估（V）、更严格的治理（G）、更丰富的可观察性（O）和更忠实的执行环境（E）通常会增加成本和延迟。速度优化通常会降低诊断深度或安全裕度，而质量优化可能会使迭代对于日常使用来说过于昂贵。真实的系统必须选择哪些检查是同步的，哪些检查是离线运行的，以及哪些故障需要昂贵的恢复路径。

标准化和生态系统动态。 MCP、ACP、A2A等协议体现了压力
面向工具、智能体 和编排状态的共享接口。  这种压力有利于智能体-不可知论者

## 11 跨层合成

本节将第 10 条的跨领域关注点整合为五个系统级影响，是权利要求 1 (§1) 的核心支持。目的是将讨论从组件覆盖转移到 Harness 行为：一旦执行、工具、上下文、编排、可观察性、评估和治理组成，它们的交互就会产生任何单层都无法单独解决的约束。

### 11.1 成本-质量-速度三难困境

Harness 可靠性受到成本、质量和速度之间三向权衡的限制。更强大的沙箱和更忠实的环境提高了安全性和可重复性，但增加了启动延迟和基础设施成本；更丰富的上下文和内存策略可以提高任务连续性，但会消耗令牌并引入检索开销；更深入的评估和可观察性可以改善诊断，但会减慢迭代速度并增加存储、标记和跟踪处理成本。因此，生产系统不能将质量视为标量目标。他们必须决定哪些风险需要昂贵的控制，哪些检查可以异步运行或在回归套件中运行，以及哪些遥测数据值得在 智能体生命周期的每个阶段捕获。

### 11.2 能力-控制权衡

能力更强的Harness 向智能体 暴露了更多的权限，但权限的每一次增加都会扩大
控制问题。  更大的工具菜单扩大了任务覆盖范围，同时增加了选择错误和提示 -
注射面；持久内存有助于长时间运行的任务，但会带来来源、过时和隐私风险；宽松的沙箱使自主执行变得有用，但也扩大了未对齐或受损操作的影响范围。因此，能力控制权衡并不是对其他功能系统的安全附加。它是一个链接工具模式、上下文策略、运行时权限、身份、可审计性和人工审批的设计轴。

### 11.3 Harness 耦合问题

Harness 层的耦合方式使得局部优化变得脆弱。执行环境通过影响包可用性、重置语义、延迟和故障模式来改变评估结果；工具描述消耗上下文预算并塑造模型行为；仅当以相同粒度捕获身份和权限状态时，可观察性痕迹才能成为治理证据；和评估设计反馈
通过奖励一些恢复循环并惩罚其他循环来进行编排。  这些耦合意味着
Harness 更改应作为系统更改进行测试。提示、工具、内存、沙箱、验证器或监视器单独看来可能是有益的，但与控制循环的其余部分结合时却会降低整个部署的性能。耦合问题还解释了为什么在不指定周围控制器的情况下，智能体 分数不能清晰地归因于模型。在闭环框架下，上下文策略、工具模式、验证器或恢复循环的更改会更改控制器 CH，从而更改同一模型 (Bölük, 2026b) 的测量行为。

### 11.4 从智能体框架到智能体平台

生态系统正在从智能体框架转向智能体平台。框架封装本地抽象，例如智能体、工具、内存存储和执行循环；平台添加了持久的工作空间、托管沙箱、身份、计费、可观察性、评估、治理以及跨多次运行和许多用户的人工切换。这种转变很重要，因为长时间运行的 智能体不再只是调用模型的程序。它们是需要租赁、合规性、故障恢复、跟踪保留和组织所有权的操作系统。因此，中心设计问题从“如何构建智能体？”转变为“如何构建智能体？” “我如何操作智能体 车队，其行为随着时间的推移保持可检查和可逆？”

<!-- page 49 -->

### 11.5 开放研究议程

综合指出了以 Harness 作为自适应控制系统为中心的研究议程。该领域需要改变 Harness 干预措施的基准，而不仅仅是模型权重、用于跨层归因故障的跟踪本机方法、用于在 智能体、工具、沙箱、评估者和人类之间转移状态和责任的协议，以及随着模型改进而简化 Harness 的优化方法。下一节将这些跨层效应转化为五个悬而未决的问题：如何强化和扩展执行环境、维护可靠状态、从跟踪诊断故障、标准化切换以及在不断变化的模型功能下保持 Harness 有用。

## 12 悬而未决的问题和未来的方向

这里收集的开放问题来自于约束约束论文和跨层综合，它们构成了权利要求 1（§1）证据的前瞻性部分。本节不是将七个 ETCLOVG 层视为独立的组件列表，而是询问整个Harness 在哪里仍然没有科学指定。中心模式是 智能体 Harness 正在成为长期运行的控制系统，但该领域仍然缺乏成熟的答案来强化执行基础、保留状态、诊断故障、转移责任以及随着模型功能的变化更新Harness。我们将这些差距整理成五个贯穿分类的问题。

### 12.1 强化和扩展执行环境

执行环境正在成为安全性、可扩展性和可移植性相结合的控制边界。 SandboxEscapeBench 证明，前沿模型可以在实际配置 (Marchand et al., 2026) 下利用沙箱的弱点，但防御工作在具有不同威胁模型和评估协议 (Wu et al., 2025; Yan, 2025) 的系统中仍然分散。与此同时，每个任务一个容器的模式给大规模训练和评估带来了压力，其中数以万计的并行轨迹需要廉价的重置和重放； SWE-World 指出了无 Docker 的代理环境，但学习到的转换相对于实际执行的保真度仍未解决(Sun et al., 2026)。甚至部署可移植性也不是一个已解决的工程细节：基于 Docker 的沙箱继承了 Linux 内核假设，而 macOS、Windows、浏览器、桌面和混合云设置则暴露出不同的隔离和可重复性约束。

悬而未决的问题是使运行时底层既可测量又可组合。未来Harness 需要针对提示注入、目标错位和成分放大进行通用安全评估；决定何时使用容器、微虚拟机、操作系统级权限边界、完整桌面虚拟机、浏览器环境或学习代理的成本模型；和可移植层，可在自托管、云和混合部署中保留语义。框架集成运行时（第 3.2.4 节）和沙箱抽象（第 3.2.7 节）之间的捆绑与组合划分应被视为经验设计问题，而不是
而非产品偏好。  MCP 等标准可能会降低合成成本，但前提是该工具，
治理层和可观察性层公开足够的状态，以保持运行时选择的可审核性、可恢复性和安全性。

### 12.2 在长时间运行中维护可靠状态智能体

最深层的上下文问题不仅仅是如何将更多标记放入提示中，而是如何保持 智能体的工作状态与长期的真实任务状态保持一致。长时间运行的编码、研究和操作智能体反复总结、检索、压缩和外部化信息；每一个这样的操作都可能删除约束、扭曲优先级或保留陈旧的假设。最近的上下文工程工作将压缩、工具结果清除、检索和提示缓存感知排序视为管理有限上下文窗口(Anthropic Applied AI Team, 2025; Anthropic, 2025c; OpenAI, 2026b)的实用机制。然而，上下文衰减和内存基准测试表明，更长的输入和更丰富的内存存储并不自动意味着更好的任务状态跟踪(Hong et al., 2025; Tan et al., 2025; He et al., 2026)。

<!-- page 50 -->

因此，有原则的研究议程应该将上下文管理重新定义为状态估计。悬而未决的问题是我们是否可以描述在每个压缩、检索或遗忘步骤中丢失了多少与任务相关的信息，以及我们是否可以限制智能体的内部状态和任务的真实状态之间的差异。最近的调查将 智能体内存形式化为写入-管理-读取循环，并将策略学习管理确定为新兴机制家族 (Zhang et al., 2025; Du, 2026)。未来的系统需要不确定性感知摘要、记忆事实的出处、矛盾处理、显式过时标记和恢复程序，让 智能体从持久工件中重建丢失的状态，而不是相信自己的压缩历史。这也表明记忆和评估之间存在更紧密的联系：记忆策略不仅应该根据回忆的准确性来判断，还应该根据它们是否可以防止多会话任务中的下游动作错误来判断。

### 12.3 从智能体 跟踪诊断故障

智能体 评估仍然常常以最终分数为中心：运行通过或失败，最终数字被视为有关模型质量的证据。对于 Harness 工程来说，这是不够的，因为失败的推出可能源于模型推理、误导性的工具模式、沙箱配置错误、陈旧的上下文、片状测试、基准模糊、判断不稳定或编排循环。 Anthropic 对智能体式 编码评估的分析表明，基础设施设置可以显着改变基准分数(Anthropic, 2026a)，而 智能体式 评估中随机性的最新研究表明，单次运行通过率可以隐藏巨大的差异(Bjarnason et al., 2026)。因此，评估层必须作为测量工具来研究，而不仅仅是用作排行榜生成器。

下一步是跟踪本机评估：跟踪应该成为系统的主要对象
计算结果分数、轨迹质量、失败归因和回归测试。可观测性系统已经捕获跨度、工具调用、成本、重试、异常和中间消息(OpenTelemetry, 2026; AlSayyad et al., 2026; Koc et al., 2025)，但这些跟踪通常与评估管道断开连接。浪链2026年调查显示，89%的团队使用可观察性，而只有52.4%的团队进行线下评估(LangChain, 2026a)；这种差距意味着团队可以看到智能体做了什么，而无需系统地判断行为是否正确。未来的工作应该通过将异常生产轨迹转换为回归案例、直接在跨度上计算轨迹指标并提供诊断信号来关闭这个循环。
最终回到提示、工具、上下文和编排更改。  反思表明智能体可以学习
来自他们自己在短视界设置中的痕迹(Shinn et al., 2023)；将这一想法扩展到长期运行、多会话Harness 仍然开放。

### 12.4 智能体、工具和人员之间的标准切换

现代 Harness 越来越多地将工作分配给规划者、子代理、工具、沙箱、评估者和人类，但这些参与者之间的接口仍然是临时的。正在出现一些本地标准：MCP 标准化工具访问，A2A 针对智能体 之间的通信，OpenTelemetry 为迹线(Model Context Protocol, 2025b; A2A Project, 2025; OpenTelemetry, 2026; Ehtesham et al., 2025) 提供通用基础。缺少的是跨层切换合约。当规划者将工作交给执行者、智能体 调用工具、子代理返回控制权或系统升级给人工时，移交不仅应传输文本摘要，还应传输意图、约束、权限、工件、出处、预算状态、风险级别、跟踪历史记录和未解决的决策。

这个问题部分是技术性的，部分是制度性的。  OpenAI 的 Symphony 构建了问题跟踪器
和存储库作为智能体工作的控制平面，而Anthropic的长期运行-智能体 Harness强调持久的进度工件和干净的切换状态(Kotliarskyi et al., 2026; Anthropic, 2025d; 2026b)。治理工作从相反的方向得出相同的结论：智能体 需要身份、委派、权限清单和可审计性，然后智能体 才能安全地代表用户跨系统(South et al., 2025; Marro et al., 2025; Syros et al., 2025) 采取行动。悬而未决的问题是定义足够丰富的安全性和恢复性的切换协议，但又足够简单以供广泛采用。此类协议应明确责任：谁授权了该操作，转移了哪些状态，哪些证据支持当前计划，允许接收者做什么，以及控制权何时必须返回到另一个智能体或人类。

### 12.5 随着模型的改进，保持Harness有用

Harness 设计不应被假设为单调地转向更多的脚手架。  每一个包装纸，
重置、验证器、规划器、内存规则和权限门编码了关于模型本身无法可靠执行哪些操作的假设。随着模型能力的变化，Harness干预措施应该重新估计，而不是假设仍然有益。按Harness 进行阶乘模型评估可以揭示干预措施何时改善所有模型、仅帮助特定模型系列或逆转模型排名(Bölük, 2026b)。 Anthropic 报告了这种模式在长期运行的应用程序开发中的具体版本：对一个模型有用的上下文重置对于更强大的模型来说变得可有可无，并且删除它们可以降低成本而不降低质量(Anthropic, 2026c)。 OpenAI 类似地将Harness 工程构建为一门保持人类注意力、存储库状态和智能体 执行一致的学科，而不是仅仅添加更多脚手架(OpenAI, 2026a;b)。

这创建了一个元工程议程：Harness 需要优化和简化自身的机制。 Meta-Harness 表明提示、工具和控制循环可以作为优化目标的一部分进行搜索，而不是手动固定 (Lee et al., 2026)，而 Natural-Language 智能体 Harness 使 Harness 模块显式且可消融 (Pan et al., 2026)。 TensorZero、Axon 和 AgentOps 等生产可观测性和成本系统指向预算敏感的 Harness 操作（TensorZero，2026；harrke-
直径177，2026； AgentOps AI，2026），但研究问题比成本最小化更广泛。  未来
系统应确定哪些干预措施对质量、安全性或可靠性负有因果关系；跨 Harness 变体运行影子模式或 A/B 测试；并在质量-延迟-成本-风险联合约束下进行优化。一个主要风险是基准过度拟合：仅针对狭窄套件进行自我优化的 Harness 可能会变得脆弱。更持久的目标是自适应简化，随着任务、工具和模型功能的变化，Harness 不断询问哪些控制仍然是必要的。

## 13 结论

这项调查将智能体 Harness视为一个独立的工程表面，并认为基础设施质量（而不仅仅是模型能力）设定了现实世界智能体可靠性的上限。围绕这个约束约束论点，我们提出了三个主张。七层 ETCLOVG 分类法将可观察性和治理与生命周期挂钩分开，并反映了生产团队如何组织其工具和所有权。将 148 多个开源项目映射到分类法中，提供了迄今为止最广泛的生态系统快照，并揭示了采用模式、覆盖范围差距和新兴设计原则。从提示到上下文再到Harness工程的三阶段工程演变，以及涵盖成本-质量-速度三难困境、能力-控制权衡和 Harness 耦合问题的跨层综合，将Harness置于更广泛的工程轨迹中。我们的分析有其局限性。该语料库偏向于英语、GitHub 可见的开源项目以及coding-智能体 生态系统；将其扩展到闭源生产系统和非编码智能体生态系统将使经验图景更加清晰。分类法本身是描述性的：将 ETCLOVG 转变为可以指导 Harness 设计决策的规范框架，而不仅仅是对它们进行分类，这是我们希望本次调查能够鼓励的自然下一步。
