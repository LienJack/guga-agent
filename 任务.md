# Guga Agent 超长任务可执行清单

> 目标：把 Guga Agent 从当前 core/plugin runtime 基线，推进到可使用、可学习、可扩展的专业 agent 平台。本文档是给 review 用的执行清单，不是最终计划；review 后再拆成 `docs/brainstorms/`、`docs/plans/`、Trellis 子任务和实现 PR。

> 最终产品目标：CLI 是基础形态；桌面端最终对齐 OpenClaw 和 Hermes Agent，成为高密度 agent workbench。本文档和 `docs/roadmap.md` 都允许在后续调研中持续修改。

## 0.1 参考仓库调研准备流程

> 适用于所有需要研究外部/参考 agent 项目的模块。目标是先把参考仓库变成 LLM 友好的研究材料，再用 `arch-insight` 做设计哲学、核心抽象、主流程、取舍和可借鉴模式分析。

- [ ] 参考仓库统一放在 `/Users/lienli/Documents/GitHub/agent-ref` 下。
- [ ] 如果调研需要一个参考仓库，而 `agent-ref` 下面还没有，可以把仓库 clone 到 `agent-ref/<repo-name>`。
- [ ] 如果参考仓库目录已存在但工作树未展开或只有 `.git`，先 fetch/checkout 默认分支，让源码 materialize 后再进入 repomix。
- [ ] 每个新增参考仓库都要记录来源、版本锚点、commit hash、调研时间和调研目的。
- [ ] 对每个参考仓库先跑 repomix token tree，产出到 `docs/research/repomix/<repo-name>-token-tree.txt`。
- [ ] 根据 token tree 选择重点路径，再跑 focused repomix context，产出到 `docs/research/repomix/<repo-name>-context*.xml` 或 `<repo-name>-focused-context.xml`。
- [ ] 对适合做概念图的仓库运行 Graphify，产出到 `docs/research/graphs/<repo-name>/graph.json` 和 `GRAPH_REPORT.md`。
- [ ] 基于 repomix / Graphify / source-analysis，整理 Context Pack，放到 `docs/research/context-packs/` 或模块专属 `docs/research/<topic>-context-pack.md`。
- [ ] Context Pack 必须包含：问题边界、参考项目与版本、必读分析材料、必读源码路径、关键抽象、已确认事实、Guga 迁移判断、待验证问题。
- [ ] 完成上述 LLM 友好材料后，再执行 `arch-insight`。
- [ ] `arch-insight` 默认输出语言为简体中文。
- [ ] 如果是为 Guga 自己做架构设计参考，`arch-insight` 默认使用 `Article - Deep Dive` 模式；如果只是快速仓库导览，才使用 `Article - Repo Overview`。
- [ ] `arch-insight` 输出必须区分：Fact、Inference、Pending Verification。
- [ ] `arch-insight` 输出必须写清：可借鉴模式、不建议照搬、Guga 落点、证据路径。
- [ ] 原始源码只在 repomix / Graphify / Context Pack 不足时打开，并且只读具体相关文件。
- [ ] OpenClaw 当前参考路径：`/Users/lienli/Documents/GitHub/agent-ref/openclaw`；当前本地目录可能需要先 checkout/materialize 源码；后续桌面应用调研必须先对它完成 repomix、Graphify、Context Pack，再用 `arch-insight` 和 Hermes/OpenCode/Claude Code 做对比。

## 1. 总体架构约束

- [ ] Core 必须保持小：只保留 runtime lifecycle、event、state、registry、hooks、permissions、projection、contracts。
- [ ] Provider、tools、context、session store、skills、MCP、memory、UI、code-agent、deep-research-agent 都必须优先通过 plugin/profile/bundle 接入。
- [ ] 专业 agent 不允许反向污染 core；code-agent 和 deep-research-agent 都应该是 first-party agent profile / plugin bundle。
- [ ] 所有副作用动作必须经过 permission runtime。
- [ ] 所有关键事实必须进入 event/audit/replay，而不是只存在于 UI 文案。
- [ ] CLI 是最基础产品形态，必须先能独立完成基础 agent 工作流。
- [ ] 桌面应用、Web、IDE/API 必须消费同一套 runtime event/protocol，不复制 agent loop。
- [ ] 每个模块都要可恢复、可审计、可回放。
- [ ] 每个模块的设计都要明确 Adopt / Adapt / Skip，不能照搬参考项目。

## 1.1 最终目标对齐

- [ ] 基础形态对齐：CLI 是 Guga 的最小可用入口和 dogfood 主入口。
- [ ] CLI 必须覆盖：interactive run、headless run、permission prompt、tool progress、artifact/diff/test output、resume、fork、cancel、debug events。
- [ ] 代码能力对齐：Claude Code 和 OpenCode。
- [ ] Code Agent 的目标是本地 coding agent，不是普通聊天 agent：读写代码、运行命令、管理 diff/test、处理权限、恢复长任务、可审计工具执行。
- [ ] 桌面应用目标对齐：OpenClaw 和 Hermes Agent。
- [ ] OpenClaw 作为桌面产品形态主参考，后续调研时必须先 materialize checkout、确认项目版本、源码锚点、产品边界和可借鉴模式。
- [ ] Hermes Agent 作为成熟长任务/网关/桌面压力样本，重点参考长任务状态面、gateway/session、权限/审批、streaming edit、工具状态和多平台投影。
- [ ] 桌面应用不是第二套 agent runtime；它是同一 runtime 的高密度工作台投影。
